The web app is a pure consumer of the API ‚Äî it has no direct database access and contains no business logic. This means IDE extensions and the CLI can use the exact same API surface.

## `packages/types`

The single source of truth for all shared shapes:

- **`PackageManifestSchema`** ‚Äî Zod schema for `ruleshub.json`, used by the API (validation on publish), the CLI (validate command), and the web (form validation)
- **`SupportedToolSchema`** ‚Äî enum of valid tool keys
- **API response types** ‚Äî `PackageDto`, `PackageVersionDto`, etc. ‚Äî shared between the web app and CLI so both stay in sync with the API

Always import from `@ruleshub/types` ‚Äî never duplicate these types in app code.

## Storage

Package zips are stored in MinIO (self-hosted S3-compatible storage). The `StorageService` in `apps/api/src/storage/` abstracts the underlying provider ‚Äî swapping to AWS S3 or Cloudflare R2 requires changing only this service.

Object keys follow the pattern: `packages/{namespace}/{name}/{version}.zip`

## Authentication

Two mechanisms, both producing a `userId` for downstream services:

1. **GitHub OAuth** ‚Äî browser flow via Passport.js, issues a JWT session cookie
2. **API keys** ‚Äî `Authorization: Bearer rh_...` header, validated by `ApiKeyGuard`

Every protected controller route has an explicit `@UseGuards(JwtAuthGuard)` or `@UseGuards(ApiKeyGuard)` ‚Äî there are no global auth defaults.

## CI/CD

- **PR ‚Üí `develop`**: lint, typecheck, e2e tests via GitHub Actions
- **Merge ‚Üí `develop`**: auto-deploy to staging (Coolify webhook)
- **Merge ‚Üí `main`**: auto-deploy to production (Coolify webhook)
